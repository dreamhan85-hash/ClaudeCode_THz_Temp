# Step 2: Multi-Agent Review Discussion

> This module is loaded from `train-orchestrator.md`. Read before executing Step 2.
> Also Read this file when used standalone from the `/review` command.

## Interface Contract

**Input** (provided by orchestrator or `/review` command):
- `experiment_n`: current experiment number
- `detail_file`: `reports/experiment_{N}_detail.md` path
- `results_file`: `results/experiment_{N}.json` (or `_aborted.json`) path
- `metric_cache`: `cache/metric_cache.jsonl` path
- `review_cycles`: number of G→round2→round3→Judge cycle repetitions (default 1)
- `config`: config file path
- `script`: training script path (and related model definition files)
- `circuit_breaker_context`: repeated pattern detection message (optional, only when circuit_breaker is active. Not provided in standalone `/review`)

**Returns**: Judge decision (`DECISION`, `RATIONALE`, `NEXT_ACTION`)

### Standalone Use (`/review` command)

When called from `/review`:
- No `session_continuation.json` manipulation
- No ralph-loop integration
- Output Judge decision directly to user
- Infer `session_dir` from the parent directory of result files

---

## Review Entry Condition

Whether training **completed normally** or was **aborted**, this step is always executed after Step 1.
- Normal completion: review based on `results/experiment_{N}.json`
- Abort: review based on `results/experiment_{N}_aborted.json` + quick reviewer abort rationale

Run the G Research Brief → round 2 → round 3 → Judge decision cycle **`{review_cycles}` times** (default: 1).

---

## Reviewer Data Delivery (File-as-IPC)

Do not insert data inline into reviewer context. Instead **pass file paths only**:
- `cache/metric_cache.jsonl`
- `results/experiment_{N}.json`
- `reports/experiment_{N}_detail.md`
- `reports/research_brief_{N}.md` (research brief generated by G — also passed to A, B, C)

Reviewers read files directly and write opinions based on numerical evidence (**800 char limit**).
Previous cycle content is already recorded in `experiment_{N}_detail.md`, reference via file path.

**All review discussion content is recorded sequentially in `reports/experiment_{N}_detail.md`.**

**Single-file principle**: do not save reviewer opinions as separate files (`cycle1_reviewer_A.md`, `cycle2_all_reviewers.md`, etc.). All review discussion content must be recorded in `experiment_{N}_detail.md` in order. The scribe does not create separate files; even if a reviewer submits a file, the scribe integrates it into `experiment_{N}_detail.md` and deletes the original.

## Research Brief (Reviewer G: Research-Informed Innovator)

Before round 2 begins, call Reviewer G via Task tool:
- **Role**: identify the core problem of the current experiment, **search for related papers/methods via WebSearch**, and propose fundamental approach changes
- **Input** (pass file paths only, no inline data):
  - `reports/experiment_{N}_detail.md` (training log + previous review records)
  - `results/experiment_{N}.json` (numerical results)
  - Current model code path (`{script}` and related model definition files)
  - `{config}`
- **Output**: write `reports/research_brief_{N}.md` file (within 1500 chars), structure:
  ```markdown
  # Research Brief — Experiment {N}
  ## Current Core Problem
  [1-2 fundamental bottlenecks identified from experiment results]
  ## Related Research and Methods
  ### Method 1: [name] ([paper/source URL])
  - Key idea: [1-2 sentences]
  - Application to our problem: [concrete approach]
  - Difference from prior attempts: [why this differs from previous failures]
  ### Method 2: ...
  ## Implementation Difficulty and Expected Impact
  [Summary of complexity and expected effect for each method]
  ```
- **Key principles**:
  - Focus on **fundamental approach changes**, not config tweaks (loss function replacement, architecture paradigm shifts, training strategy changes, etc.)
  - When proposing, always explain **"why it works under our data/model's specific conditions"**
  - G does not participate in the consensus round and does not retract proposals
  - Whether A, B, C adopt, reject, or modify G's proposals is each reviewer's own judgment
- **Returns**: only `"brief complete"` to orchestrator

**Call scribe**: record `research_brief_{N}.md` creation in `experiment_{N}_detail.md`. Returns: only `"recorded"`.

※ G's research brief is passed to A, B, C as a file path for **reference**. G's proposed methods do not replace the independent judgment of A, B, C.

## Round 2

**Step 1 (parallel)**
Call Reviewers A, B, C simultaneously via Task tool:
- Reviewer A: quantitative/statistical perspective (loss curve, overfitting, metric trends, direct citation of results)
- Reviewer B: algorithm/model architecture perspective (model design, training strategy appropriateness)
- Reviewer C: data structure/design perspective (data pipeline, preprocessing, data quality, **figure quality**: axis ranges, label overlap, legend placement, annotation readability, font size, sequential numbering)
- ※ All of A, B, C receive G's `reports/research_brief_{N}.md` as reference. Each decides whether to adopt it.

**Step 2 (parallel)** — after A, B, C results
Call Reviewers D, E simultaneously via Task tool:
- Reviewer D (Feasibility Assessor): evaluate **failure scenarios for each proposal from A, B, C**. Not simple rebuttals but "under what conditions would this proposal fail". Rate risk for each proposal as **[LOW/MEDIUM/HIGH]**, and for HIGH-risk proposals provide **actionable conditions** (under what assumptions it's worth trying). Evaluate G's research brief proposals with the same criteria.
- Reviewer E (Supplement): fill in missing points from A, B, C opinions, supplement supporting evidence

**Step 3 (sequential)** — after A–E results
Call Reviewer F via Task tool:
- Reviewer F (Moderator): clearly summarize consensus and conflict points across all A–E opinions. Draw the landscape only, without making judgments.

**Call scribe**: pass file path (`reports/experiment_{N}_detail.md`) + 1-line summary per reviewer decision. Scribe collects reviewer opinion details directly from files. Returns: only `"recorded"`.

---

## Round 3 (parallel) — after F results

Call Reviewers A, B, C simultaneously via Task tool:
- Review D, E, F opinions and submit updated positions maintaining or revising their original arguments
- Explain reasons when changing position

**Call scribe**: pass file path + 1-line summary per reviewer decision. Scribe collects directly from files. Returns: only `"recorded"`.

---

## Judge (Judge Agent)

After `{review_cycles}` complete, the orchestrator does not make the decision directly but calls a **separate Judge agent** via Task tool:
- **Purpose**: without contaminating orchestrator's context, synthesize review discussion objectively in a fresh context
- **Input** (pass file paths only):
  - `reports/experiment_{N}_detail.md` (full review discussion)
  - `results/experiment_{N}.json` (numerical results)
  - `cache/metric_cache.jsonl` (per-epoch metrics)
  - `reports/research_brief_{N}.md` (G's research brief)
- **Decision criteria**: apply the orchestrator's **decision-making criteria** as-is (including Strict Convergence and Bold Improvement principles)
- **Special rules when deciding**:
  - G's proposals not adopted by A, B, C may still be reflected in `algo_modify` decisions if well-supported
  - Even proposals rated HIGH risk by D can be adopted if actionable conditions are clear
  - Prioritize **quality of evidence** over level of reviewer consensus
  - If `circuit_breaker_context` is provided: must mention it and specify NEXT_ACTION that breaks away from the repeated pattern
    - `config_modify` repeat pattern: maintain only if justification is very strong. Otherwise strongly recommend `algo_modify`
    - `algo_modify` repeat pattern: specify in NEXT_ACTION what assumption was wrong. Include "user review recommended" at end of NEXT_ACTION
    - circuit_breaker_context is contextual information; same-type decision can be maintained if evidence is sufficient, but reason must be stated in RATIONALE
- **Returns**: structured decision (loaded into orchestrator context):
  ```
  DECISION: [go / config_modify / algo_modify / abort]
  RATIONALE: [rationale within 200 chars]
  NEXT_ACTION: [next experiment plan — include specific changes for config/algo modifications]
  ```
- Orchestrator **accepts the Judge's decision as-is** and proceeds to execution
- **Call scribe**: pass file path + full Judge decision. Scribe records directly. Returns: only `"recorded"`.

※ The Judge agent is not a reviewer and does not participate in discussion. It makes **1 decision** after the review discussion is complete.
